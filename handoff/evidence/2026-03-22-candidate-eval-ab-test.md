# Handoff A/B Test: candidate-eval (2026-03-22)

Real-world comparison of `/grill-me` sessions with and without a handoff document, run back-to-back on the same codebase with the same prompt.

## Setup

- **Project:** candidate-eval (privacy-first job fit assessment pipeline)
- **Codebase:** ~10,700 LOC Python, 931 tests, Chrome extension, FastAPI server
- **Prompt:** `/grill-me` → "next steps and current implementation and user flow"
- **Model:** Claude Opus 4.6 (1M context) with high effort
- **Branch:** `main` after merging PRs #10 and #11 (extractor overhaul + merge pipeline fixes)

## Session A: With Handoff

```
claude --append-system-prompt-file .claude/handoffs/latest.md --dangerously-skip-permissions
```

Handoff document: ~850 tokens covering summary, 6 decisions, 5 do-not-retry items, 4 constraints, 6 next steps, 7 key files.

### Exploration Phase
- **2 Explore agents** dispatched
- **9 total tool uses** (6 + 3)
- **~56k tokens** consumed exploring
- **4 files read** — only the files needed to verify handoff claims
- **~2 minutes** to first grilling question

### First Question
> "You have a tool that requires — before the browser extension does anything useful — that the user has: (1) installed the Python package, (2) scanned their sessions with a whitelist, (3) either parsed a resume or curated one, (4) merged profiles, and (5) started the server. That's at least 5 CLI commands with correct arguments before a single LinkedIn assessment works. What happens when someone installs this for the first time and clicks the extension icon?"

Went straight to a high-value UX gap (onboarding friction) because it already knew the architecture and could focus on user flow analysis.

## Session B: Without Handoff

```
claude --dangerously-skip-permissions
```

No handoff, no system prompt injection. Same `/grill-me` → same prompt.

### Exploration Phase
- **2 Explore agents** dispatched
- **83 total tool uses** (38 + 45)
- **~197k tokens** consumed exploring
- **30+ files read** — every schema, module, memory file, plan, CLI command, server endpoint, extension file
- **~4 minutes** to first grilling question
- Hit file size limits multiple times, had to retry with offset/limit

### First Question
> "You call this a 'portfolio piece, not a product.' But the codebase has SQLite caching, a Chrome extension with dashboard and shortlisting, a FastAPI server, batch assessment, fire-and-forget background jobs, and a delivery pipeline to roojerry.com. That's product-level infrastructure for something you claim isn't a product. What is the actual end state you're building toward?"

Also a great question — but it took 4x the resources and 2x the time to get there.

## Quality Assessment

Both sessions produced high-quality risk trees:

**Session A** focused on user onboarding (Branch 1.1 — practical UX friction) because it already knew the architecture and could go straight to gap analysis.

**Session B** focused on identity/end-state (Branch 1.1 — strategic clarity) because it had to build the full mental model from scratch and naturally started with "what is this?"

Both are valid starting points. The difference is cost, not quality. Session A got to the same caliber of questioning with **72% fewer tokens**, preserving context window budget for the actual deep grilling that follows.

## Why This Matters

The handoff is ~850 tokens — less than 0.1% of the context window. It saves ~140k tokens of exploration per session. For a project with ongoing multi-session development, that compounds:

- **10 sessions** = ~1.4M tokens saved
- **Context window preservation** = deeper conversations before compression kicks in
- **Negative knowledge transfer** = fresh sessions don't re-attempt failed approaches (e.g., corroboration depth boost, fuzzy matching on individual words, confidence penalties on resume-only skills)

The last point is hardest to quantify but arguably most valuable. Without the handoff, Session B would have no way to know that "experience" fuzzy-matches to "startup-experience" or that the benchmark uses a different code path than the CLI — knowledge that took hours of debugging to discover.
